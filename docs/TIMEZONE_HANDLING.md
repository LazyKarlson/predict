# 🌍 Работа с часовыми поясами

## Проблема

Олимпийские игры 2026 проходят в Милане/Кортина д'Ампеццо (Италия, UTC+1).
Участники конкурса могут быть из разных часовых поясов:
- 🇷🇺 Москва (UTC+3)
- 🇺🇸 Нью-Йорк (UTC-5)
- 🇺🇸 Лос-Анджелес (UTC-8)
- 🇯🇵 Токио (UTC+9)
- и т.д.

**Критично**: Пользователь не должен иметь возможность сделать ставку после начала матча, независимо от его часового пояса.

## Решение

### 1. Хранение времени в UTC

Все времена начала матчей хранятся в базе данных в **UTC** (Coordinated Universal Time).

```sql
CREATE TABLE matches (
    ...
    match_date DATETIME NOT NULL, -- Хранится в UTC
    ...
);
```

### 2. Конвертация при вводе (Админ)

Когда админ создает матч:

```javascript
// Frontend (views/admin/matches.ejs)
const localDateTime = formData.get('match_date'); // "2026-02-15T15:00"
const localDate = new Date(localDateTime);        // Локальное время браузера
const utcDateTime = localDate.toISOString();      // "2026-02-15T12:00:00.000Z" (UTC)
```

**Пример:**
- Админ в Москве (UTC+3) вводит: `15 февраля 2026, 15:00`
- Система конвертирует в UTC: `15 февраля 2026, 12:00 UTC`
- Сохраняется в БД: `2026-02-15 12:00:00`

### 3. Отображение времени (Пользователи)

Каждый пользователь видит время в своем локальном часовом поясе:

```javascript
// Frontend (views/index.ejs)
new Date(match.match_date).toLocaleString('ru-RU', {
    day: 'numeric',
    month: 'long',
    hour: '2-digit',
    minute: '2-digit'
});
```

**Пример для матча `2026-02-15 12:00:00 UTC`:**
- 🇷🇺 Пользователь в Москве видит: `15 февраля, 15:00`
- 🇺🇸 Пользователь в Нью-Йорке видит: `February 15, 07:00`
- 🇺🇸 Пользователь в Лос-Анджелесе видит: `February 15, 04:00`
- 🇯🇵 Пользователь в Токио видит: `2月15日, 21:00`

### 4. Проверка времени (Backend)

Проверка происходит на сервере в UTC:

```typescript
// Backend (src/routes/api.ts)
const matchDate = new Date(match.match_date); // UTC из БД
const now = new Date();                       // Текущее UTC время сервера

if (matchDate <= now) {
    return res.status(400).json({ 
        error: 'Матч уже начался. Прогнозы больше не принимаются.' 
    });
}
```

## Примеры работы

## Технические детали

### JavaScript Date и часовые пояса

```javascript
// Создание даты из строки datetime-local
const localStr = "2026-02-15T15:00";
const date = new Date(localStr);
// date хранит время в UTC, но интерпретирует строку как локальное время

// Конвертация в UTC ISO строку
const utcStr = date.toISOString();
// "2026-02-15T12:00:00.000Z" (если локальный UTC+3)

// Отображение в локальном времени
const displayStr = date.toLocaleString('ru-RU');
// "15.02.2026, 15:00" (в локальном часовом поясе)
```

### MySQL DATETIME и часовые пояса

```sql
-- DATETIME хранит время "как есть" без информации о часовом поясе
-- Мы договариваемся, что это всегда UTC

INSERT INTO matches (match_date) VALUES ('2026-02-15 12:00:00');
-- Сохраняется: 2026-02-15 12:00:00 (мы знаем, что это UTC)

SELECT match_date FROM matches WHERE id = 1;
-- Возвращается: 2026-02-15 12:00:00 (мы интерпретируем как UTC)
```

### Node.js и часовые пояса

```typescript
// new Date() всегда возвращает текущее UTC время
const now = new Date();
console.log(now.toISOString()); // "2024-02-05T10:30:00.000Z"

// Сравнение дат происходит в миллисекундах с Unix Epoch (UTC)
const matchDate = new Date('2026-02-15T12:00:00.000Z');
if (matchDate <= now) {
    // Корректное сравнение в UTC
}
```

## Рекомендации для админа

### При создании матчей:

1. **Узнайте официальное время начала матча** (обычно указано в расписании Олимпиады)
2. **Конвертируйте в свой часовой пояс** (если нужно)
3. **Введите время в форму** - система автоматически конвертирует в UTC

### Пример:

Официальное расписание: `Матч Россия - Канада, 15 февраля 2026, 20:00 CET (Милан)`

**Если вы в Москве (UTC+3):**
- CET = UTC+1
- Разница: Москва на 2 часа впереди
- Вводите: `15 февраля 2026, 22:00`

**Если вы в Нью-Йорке (UTC-5):**
- CET = UTC+1
- Разница: Нью-Йорк на 6 часов позади
- Вводите: `February 15, 2026, 14:00`

## Часто задаваемые вопросы

**Q: Что если сервер находится в другом часовом поясе?**  
A: Не важно. `new Date()` в Node.js всегда возвращает UTC время.

**Q: Что если у пользователя неправильно настроено время?**  
A: Проверка происходит на сервере. Время клиента используется только для отображения.

**Q: Нужно ли настраивать часовой пояс на сервере?**  
A: Нет. Вся логика работает в UTC независимо от настроек сервера.

**Q: Как быть с переходом на летнее время?**  
A: UTC не имеет перехода на летнее время. Все работает корректно автоматически.
