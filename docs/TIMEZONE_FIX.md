# Исправление обработки часовых поясов

## Проблема

Пользователи видели, что матчи отображаются как "уже начавшиеся", хотя по их локальному времени матч еще не начался.

**Пример:**
- Матч начинается в 18:40 по московскому времени (UTC+3)
- В базе данных хранится: `2026-02-11 15:40:00` (UTC)
- Пользователь видел: "Матч уже начался" в 18:40 по своему времени

## Причина

MySQL возвращает DATETIME как строку в формате `YYYY-MM-DD HH:MM:SS` **без указания часового пояса**.

JavaScript интерпретирует такую строку как **локальное время**, а не UTC:

```javascript
// ❌ НЕПРАВИЛЬНО
const date = new Date('2026-02-11 15:40:00'); 
// JavaScript думает, что это 15:40 по локальному времени

// ✅ ПРАВИЛЬНО
const date = new Date('2026-02-11T15:40:00Z'); 
// JavaScript понимает, что это 15:40 UTC
```

## Решение

### 1. Серверная часть (`src/routes/pages.ts`)

Конвертируем `match_date` из MySQL Date объекта в ISO строку с 'Z' суффиксом:

```typescript
const matchesWithUTC = matches.map((match: any) => ({
    ...match,
    match_date: match.match_date.toISOString() // '2026-02-11T15:40:00.000Z'
}));
```

### 2. Клиентская часть (`views/index.ejs`)

Теперь `match.match_date` приходит как ISO строка с 'Z', JavaScript правильно интерпретирует её как UTC:

```javascript
const matchDate = new Date(match.match_date); // Правильно парсит UTC
const now = new Date(); // Текущее время в UTC
const isStarted = matchDate <= now; // Корректное сравнение
```

### 3. Отображение времени

`toLocaleString()` автоматически конвертирует UTC время в локальный часовой пояс пользователя:

```javascript
matchDate.toLocaleString('ru-RU', {
    day: 'numeric',
    month: 'long',
    hour: '2-digit',
    minute: '2-digit'
})
// Для пользователя в UTC+3: "11 февраля, 18:40"
// Для пользователя в UTC+0: "11 февраля, 15:40"
```

## Результат

✅ Время в базе данных остается в UTC  
✅ Сервер корректно проверяет, начался ли матч (UTC vs UTC)  
✅ Клиент корректно проверяет, начался ли матч (UTC vs UTC)  
✅ Пользователь видит время в своем локальном часовом поясе  
✅ Прогнозы блокируются в правильное время для всех часовых поясов

2. **`views/index.ejs`** - упрощена обработка даты (теперь приходит правильный формат)

## Тестирование

### Проверка отображения времени

1. Открой главную страницу
2. Проверь, что время матчей отображается в твоем локальном часовом поясе
3. Если ты в UTC+3, матч в 15:40 UTC должен показываться как 18:40

### Проверка блокировки прогнозов

1. Попробуй сделать прогноз на матч, который еще не начался по твоему времени
2. Должно работать ✅
3. Попробуй сделать прогноз на матч, который уже начался по твоему времени
4. Должна появиться ошибка "Матч уже начался" ❌

### Проверка в разных часовых поясах

Можно эмулировать разные часовые пояса в Chrome DevTools:
1. F12 → Console → Settings (⚙️) → Show sensors
2. Location → выбери другой город
3. Перезагрузи страницу
4. Время должно отображаться в выбранном часовом поясе

## Важные моменты

⚠️ **Всегда храни время в UTC в базе данных**
⚠️ **Всегда передавай время с явным указанием часового пояса (ISO с 'Z')**
⚠️ **Используй `toLocaleString()` для отображения в локальном времени**
⚠️ **Сравнивай время только в одном часовом поясе (UTC vs UTC)**

## Безопасность

### Можно ли обойти проверку времени, изменив системное время?

**Нет!** Есть две линии защиты:

1. **Клиентская проверка (в браузере)**
   - Использует системное время пользователя
   - Можно обойти, изменив время в системе
   - Служит только для UX (скрывает форму)

2. **Серверная проверка (в API)** ✅
   - Использует время сервера (не зависит от клиента)
   - **Невозможно обойти** изменением времени на компьютере пользователя
   - Проверяет время перед сохранением прогноза
   - Код: `src/routes/api.ts`, строка 201

**Пример:**
```
Пользователь изменил время на своем компьютере на 17:00
→ Форма прогноза отображается (клиентская проверка обманута)
→ Пользователь отправляет прогноз
→ Сервер проверяет время по своим часам (18:30)
→ Сервер отклоняет прогноз: "Матч уже начался" ✅
```

### Дополнительная защита

Также проверяется флаг `is_finished`:
- Если матч завершен, прогнозы не принимаются
- Даже если время сервера неправильное

## Дополнительные исправления

### CSRF и CSP проблемы

При реализации исправлений возникли проблемы с безопасностью:

1. **CSP блокировал inline event handlers**
   - Проблема: `onsubmit="savePrediction(event, matchId)"` нарушал CSP
   - Решение: Переписали на `addEventListener` в JavaScript
   - Файл: `views/index.ejs`

2. **CSRF cookie не устанавливалась**
   - Проблема: Префикс `__Host-` требует HTTPS, не работает на localhost
   - Решение: Убрали префикс, используем `x-csrf-token`
   - Файл: `src/server.ts`
   - В production рекомендуется вернуть `__Host-` префикс при использовании HTTPS
